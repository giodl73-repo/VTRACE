# The V Shape For Software

The left side of the V defines intent. The bottom implements. The right side
proves that the implementation satisfies the intent.

| Left Side | Bottom / Right Side |
|---|---|
| Mission and need | Validation against intended use |
| CONOPS scenarios | Acceptance scenarios and walkthroughs |
| Requirements | System verification |
| Specification baseline | Product verification |
| Architecture and package boundaries | Integration verification |
| Interfaces | Interface verification |
| Design and code rigor | Component, unit, static, and review evidence |
| Work packages | Controlled implementation and closeout evidence |

The practical rule is that every left-side claim should know what evidence will
close it on the right side. If the evidence cannot be named, the claim is not
ready for implementation or needs an accepted deferral.
